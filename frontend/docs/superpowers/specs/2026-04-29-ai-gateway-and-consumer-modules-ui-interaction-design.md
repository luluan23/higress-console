# Higress Console AI 网关管理与消费者管理模块 UI 交互设计文档

## 1. 文档目标

本文档面向 UI 交互设计师，作为以下 3 个模块的页面设计基线：

1. AI 服务提供者管理
2. AI 路由管理
3. 消费者管理

文档重点描述页面目标、信息架构、页面组成、核心交互、状态覆盖和跨页面关系，设计师可据此直接开展页面框架、交互流程和状态稿设计。

## 2. 设计范围

本次设计范围包含以下页面与直接衍生交互：

1. AI 服务提供者管理列表页
2. AI 服务提供者新建/编辑抽屉
3. AI 路由管理列表页
4. AI 路由新建/编辑抽屉
5. AI 路由使用方法弹窗
6. AI 路由策略配置入口与列表展开区
7. 消费者管理列表页
8. 消费者新建/编辑抽屉

以下相邻模块仅作为依赖对象在文中提及，不展开设计：

1. 域名管理
2. AI 仪表盘
3. 通用路由管理
4. 插件管理
5. 服务来源管理

## 3. 业务对象与模块关系

### 3.1 三个核心对象

| 对象 | 业务角色 | 设计关注点 |
| --- | --- | --- |
| AI 服务提供者 | 定义网关连接的 AI 上游服务 | 上游类型、接入方式、凭证、代理、服务端点摘要 |
| AI 路由 | 定义 AI 请求入口、匹配条件、转发目标和授权对象 | 匹配规则、目标服务、降级策略、请求授权、策略入口 |
| 消费者 | 定义调用方身份与认证信息 | 消费者名称、认证方式、凭证录入与来源 |

### 3.2 对象关系

| 关系 | 说明 |
| --- | --- |
| AI 服务提供者 -> AI 路由 | AI 路由的主服务和降级服务都从 AI 服务提供者中选择 |
| 消费者 -> AI 路由 | 当 AI 路由开启请求认证时，从消费者中选择允许访问的对象 |
| 域名 -> AI 路由 | AI 路由可绑定一个或多个域名 |

对设计师而言，AI 路由页是三个对象的汇合点，也是信息密度最高的页面。

## 4. 整体设计框架

### 4.1 页面骨架

三个模块采用统一的页面结构：

1. 顶部操作区或筛选区
2. 主表格区
3. 右侧抽屉，用于新建和编辑
4. 删除确认弹窗，用于危险操作

### 4.2 创建态与编辑态

三个模块均采用“同一个抽屉承载创建与编辑”的模式。

编辑态下，作为稳定标识的名称类字段需要锁定：

1. AI 服务提供者：供应商类型、服务名称不可修改
2. AI 路由：路由名称不可修改
3. 消费者：消费者名称不可修改

### 4.3 列表页共性

1. 资源默认按名称排序展示
2. 长文本根据内容类型采用省略、换行或标签集合的方式展示
3. 表格操作区以行级操作为主，不涉及批量操作

### 4.4 全局状态覆盖

三个模块的设计稿都需要覆盖以下状态：

1. 列表加载中
2. 列表为空
3. 有数据的正常列表
4. 抽屉初始态
5. 表单校验态
6. 删除确认态
7. 请求失败后的反馈态

## 5. AI 服务提供者管理

### 5.1 页面目标

AI 服务提供者管理页用于维护可被 AI 路由引用的上游 AI 服务。页面重点是帮助用户完成以下任务：

1. 创建新的 AI 服务提供者
2. 查看已有提供者的基础摘要
3. 编辑提供者的接入配置
4. 删除不再使用的提供者

### 5.2 页面结构

页面由 4 个交互层构成：

1. 顶部操作条
2. 提供者列表表格
3. 新建/编辑抽屉
4. 删除确认弹窗

### 5.3 顶部操作条

顶部区域包含两个核心动作：

1. 主按钮：创建 AI 服务提供者
2. 次按钮：刷新

该页面不包含搜索、筛选和分页，因此视觉重点应放在主按钮与表格信息本身。

### 5.4 列表页信息设计

表格包含以下列：

| 列名 | 展示内容 | 设计说明 |
| --- | --- | --- |
| 类型 | 提供者类型名称 | 以统一命名体系展示，不直接暴露内部表达 |
| 名称 | 服务名称 | 作为资源主名称展示 |
| 服务端点 | 当前提供者实际连接的服务入口 | 可能为单值，也可能为多值换行 |
| 凭证 | 凭证摘要 | 默认隐藏，支持逐条切换显示/隐藏 |
| 代理 | 代理名称 | 无代理时显示占位符 |
| 操作 | 编辑、删除 | 行内操作入口 |

#### 凭证列交互

凭证区的设计重点是“默认保护敏感信息，但保留人工核对能力”：

1. 默认隐藏
2. 支持逐条切换显示/隐藏
3. 多个凭证需要清晰分行

#### 服务端点列展示特点

不同类型的提供者，服务端点的展示形态不同，因此该列更适合设计为“结果摘要型”信息，而不是原始字段回显：

1. 可能是固定官方地址
2. 可能是一个自定义地址
3. 可能是多个地址并列展示
4. 可能是通过内部服务拼出的访问地址

### 5.5 新建/编辑抽屉

抽屉为中宽规格，适合承载长表单。标题根据模式切换为“创建 AI 服务提供者”或“编辑 AI 服务提供者”。

建议设计上将内容分为 4 个信息区块：

1. 基础信息
2. 凭证配置
3. 连接配置
4. 高级配置

#### 基础信息

基础信息区包含：

1. 大模型供应商
2. 服务名称
3. 协议

其中供应商和服务名称在编辑态下为只读状态。

#### 凭证配置

凭证区需要支持动态增删，单个提供者可配置多个凭证。该区需覆盖以下交互：

1. 新增一条凭证
2. 删除一条凭证
3. 多条凭证并列编辑
4. 部分供应商使用自定义凭证布局
5. 展示“引用 Secret 中的凭证信息”的说明入口

“引用 Secret 中的凭证信息”是说明性入口，应设计为轻量帮助触发器，点击后打开说明弹窗。

#### 连接配置

连接配置区包含：

1. 代理选择
2. 与供应商类型相关的接入字段

这是整个表单中变化最多的区域，需根据供应商类型切换不同字段组合。

#### 高级配置

高级配置区用于承载“令牌降级”相关设置。交互方式为：

1. 先通过开关决定是否启用
2. 开启后展开详细字段

展开后需要展示一组连续的高级参数，整体应设计成一块独立配置区，而不是分散插入表单中。

### 5.6 供应商类型驱动的表单变化

#### OpenAI

OpenAI 需要设计成多层选择流程：

1. 先选择官方服务或自定义服务
2. 若选择自定义服务，再选择“输入地址”或“选择服务”

因此 OpenAI 至少需要覆盖 3 种表单状态：

1. 官方服务状态
2. 自定义地址状态
3. 选择服务状态

#### Qwen

Qwen 需要展示以下可配置内容：

1. 联网搜索开关
2. 兼容模式开关
3. 文件 ID 列表
4. 官方服务与自定义域名切换

#### Azure

Azure 以单一服务地址为核心配置项，页面重点在于单字段的清晰呈现和错误提示空间。

#### 其他主要类型

其他类型需要按各自关键配置设计独立字段块：

| 类型 | 主要配置内容 |
| --- | --- |
| Zhipu AI | 自定义域名、Code Plan 模式 |
| Claude | API 版本、Claude Code 模式 |
| Ollama | 主机地址、端口 |
| Bedrock | 区域、访问凭证 |
| Vertex AI | 区域、项目 ID、服务账号信息、安全设置 |
| vLLM | 一个或多个自定义地址 |

### 5.7 删除确认弹窗

删除交互采用二次确认弹窗。弹窗重点是：

1. 明确当前要删除的提供者名称
2. 强调这是危险操作
3. 提供取消与确认删除两个动作

### 5.8 设计稿输出清单

AI 服务提供者模块至少需要输出以下页面与状态稿：

1. 列表默认态
2. 列表空状态
3. 凭证隐藏态与显示态
4. 创建抽屉初始态
5. 编辑抽屉态
6. OpenAI 三种表单状态
7. Qwen 表单状态
8. Azure 表单状态
9. 令牌降级关闭态与展开态
10. Secret 说明弹窗
11. 删除确认弹窗

## 6. AI 路由管理

### 6.1 页面目标

AI 路由管理页是 AI 服务配置的中枢页，用于把请求入口、匹配规则、目标服务、授权对象和策略入口组织到同一个资源中。页面重点是帮助用户完成以下任务：

1. 创建新的 AI 路由
2. 查看路由摘要信息
3. 配置或修改匹配规则、转发目标和授权对象
4. 查看调用示例
5. 进入策略配置
6. 删除路由

### 6.2 页面结构

页面包含 6 个交互层：

1. 顶部查询与操作区
2. 路由主表格
3. 行展开区，用于展示策略摘要
4. 新建/编辑抽屉
5. 使用方法弹窗
6. 删除确认弹窗

### 6.3 顶部查询与操作区

顶部区域包含：

1. 搜索框
2. 创建 AI 路由按钮
3. 刷新按钮

搜索框用于模糊搜索以下信息：

1. 路由名称
2. 域名
3. 路径匹配值
4. 上游服务名称

### 6.4 列表页信息设计

主表格包含以下列：

| 列名 | 展示内容 | 设计说明 |
| --- | --- | --- |
| 名称 | 路由名称 | 资源主名称 |
| 域名 | 绑定域名列表 | 多值换行展示 |
| 路径匹配规则 | 匹配方式与匹配值 | 摘要型表达 |
| 模型匹配规则 | 模型匹配列表 | 多值换行展示 |
| 服务 | 主服务与降级服务摘要 | 单值与多值两种表达 |
| 请求授权 | 认证状态与授权对象摘要 | 需要区分多种状态 |
| 操作 | 使用方法、策略、编辑、删除 | 多动作入口 |

#### 服务列展示规则

服务列承担“主服务 + 降级服务”的压缩展示任务：

1. 只有一个主服务时，直接展示服务名称
2. 存在多个主服务时，需要同时体现服务名称与权重
3. 存在降级服务时，需要在视觉上区分主服务与降级服务

#### 请求授权列展示规则

请求授权列至少需要覆盖 3 种状态：

1. 未开启认证
2. 已开启认证但未选择任何消费者
3. 已开启认证且已选择消费者

当消费者数量较多时，列表中以摘要方式展示，完整名单通过悬浮层查看。

### 6.5 行展开区

表格行支持展开。展开后展示当前路由已启用的策略清单，信息包括：

1. 策略名称
2. 策略描述

该区域适合作为嵌套信息区，帮助用户在不跳页的情况下快速判断这条路由是否挂载了额外策略。

### 6.6 列表操作区

#### 使用方法

“使用方法”通过弹窗展示当前路由的调用示例。设计上应容纳以下内容：

1. 请求入口示例
2. 域名相关提示
3. 基础请求示例

#### 策略

“策略”是一个跳转型入口，用于进入该路由的策略配置页。

#### 编辑与删除

编辑打开右侧抽屉，删除打开确认弹窗。

### 6.7 新建/编辑抽屉

AI 路由抽屉是信息最复杂的表单，建议设计为分段明显的长表单。推荐按以下顺序组织：

1. 基础标识
2. 路径匹配
3. Header 匹配
4. Query 匹配
5. 模型匹配
6. 目标 AI 服务
7. 降级配置
8. 请求认证
9. 附加注解

#### 基础标识

基础标识区包含：

1. 路由名称
2. 域名选择

域名为多选项，并带有刷新入口。默认系统域名不出现在可选项中。

#### 路径匹配

路径匹配区由以下元素组成：

1. 匹配方式
2. 匹配值
3. 是否忽略大小写

当前基线中，路径匹配采用前缀匹配形式。

#### Header 匹配与 Query 匹配

这两块均为可重复规则列表。每条规则包含：

1. 名称
2. 匹配方式
3. 匹配值
4. 删除动作

匹配方式支持精确匹配、前缀匹配和正则匹配。

#### 模型匹配

模型匹配用于在命中路由后继续按模型名细分。每条规则包含：

1. 匹配方式
2. 匹配值

#### 目标 AI 服务

这是 AI 路由表单中的核心区块。每个目标服务条目包含：

1. AI 服务提供者选择
2. 权重
3. 目标模型映射

该区需要支持：

1. 新增一个目标服务条目
2. 删除一个目标服务条目
3. 多个服务条目并列配置
4. 模型映射的简洁输入与规则化编辑两种入口

视觉上建议把每个目标服务条目设计成结构清晰的独立行或独立卡片。

#### 降级配置

降级配置由一个开关控制，开启后展示完整配置区。该区包含：

1. 触发降级的响应码
2. 降级服务选择
3. 降级模型映射

降级服务为单选。

#### 请求认证

请求认证同样由开关控制。开启后展示：

1. 认证类型
2. 允许访问的消费者列表

当前基线中，认证类型固定为 Key Auth，应以只读或禁用态展示；消费者为多选，并带有刷新入口。

#### 附加注解

附加注解采用键值对列表形式，用于承载额外说明性配置。该区应提供：

1. 新增一组键值对
2. 删除一组键值对
3. 外部帮助入口

### 6.8 删除确认弹窗

删除确认弹窗需要明确展示：

1. 当前要删除的路由名称
2. 危险操作提示
3. 取消与确认删除按钮

### 6.9 设计稿输出清单

AI 路由模块至少需要输出以下页面与状态稿：

1. 列表默认态
2. 搜索结果态
3. 行展开态
4. 使用方法弹窗
5. 创建抽屉初始态
6. 编辑抽屉态
7. 多个目标服务并列配置态
8. 降级配置关闭态与展开态
9. 请求认证关闭态与展开态
10. 已开启认证但未选择消费者的状态稿
11. 附加注解编辑态
12. 删除确认弹窗

## 7. 消费者管理

### 7.1 页面目标

消费者管理页用于维护调用方身份及其认证信息，供 AI 路由在开启请求认证时引用。页面重点是帮助用户完成以下任务：

1. 创建新的消费者
2. 查看消费者的认证方式摘要
3. 编辑认证信息
4. 删除消费者

### 7.2 页面结构

页面由 4 个交互层组成：

1. 顶部筛选与操作区
2. 主表格
3. 新建/编辑抽屉
4. 删除确认弹窗

相较另外两个模块，消费者管理页还包含分页区。

### 7.3 顶部筛选与操作区

顶部区域包含：

1. 按消费者名称筛选
2. 按密钥关键字筛选
3. 重置
4. 新建消费者
5. 刷新

整体适合设计为一行内联筛选条，左侧为筛选项，右侧为动作按钮。

### 7.4 列表页信息设计

主表格包含以下列：

| 列名 | 展示内容 | 设计说明 |
| --- | --- | --- |
| 消费者名称 | 消费者名称 | 资源主名称 |
| 认证方式 | 当前已配置的认证方式摘要 | 以标签形式展示 |
| 操作 | 编辑、删除 | 行内操作 |

认证方式列强调“方式摘要”而不是“凭证细节”，因此更适合用标签形式表达。

### 7.5 分页

消费者列表需要支持分页，分页区需覆盖：

1. 页码切换
2. 每页条数切换
3. 总数展示

### 7.6 新建/编辑抽屉

消费者抽屉同样采用中宽规格，标题根据模式在“创建消费者”和“编辑消费者”之间切换。

#### 基础信息

基础信息区包含：

1. 消费者名称

编辑态下名称不可修改。

#### 认证方式切换

抽屉主体采用标签页形式组织认证方式，当前包含：

1. Key Auth
2. OAuth2
3. JWT

其中，Key Auth 为实际可编辑内容区；OAuth2 和 JWT 为预留标签页，需要保留其占位状态设计。

#### Key Auth 配置区

Key Auth 区包含以下交互：

1. 动态凭证列表
2. 单条凭证的一键随机生成
3. 凭证来源选择
4. 随来源切换而变化的附加字段

凭证来源包含 3 种：

1. Bearer
2. Header
3. Query

对应的界面状态如下：

1. 选择 Bearer 时，不展示额外名称字段
2. 选择 Header 时，展示 Header 名称字段
3. 选择 Query 时，展示参数名称字段

### 7.7 删除确认弹窗

删除确认弹窗需要明确展示：

1. 当前要删除的消费者名称
2. 危险操作提示
3. 取消与确认删除按钮

### 7.8 设计稿输出清单

消费者模块至少需要输出以下页面与状态稿：

1. 列表默认态
2. 筛选结果态
3. 空状态
4. 分页态
5. 创建抽屉初始态
6. 编辑抽屉态
7. Key Auth 下的 Bearer 状态
8. Key Auth 下的 Header 状态
9. Key Auth 下的 Query 状态
10. OAuth2 占位标签页态
11. JWT 占位标签页态

## 8. 跨页面流程与依赖关系

### 8.1 典型使用链路

从业务流程上，用户通常会按照以下顺序完成配置：

1. 先创建 AI 服务提供者
2. 再创建消费者
3. 再进入 AI 路由管理创建路由
4. 在路由中绑定域名、匹配规则、目标服务和授权对象
5. 保存后查看使用方法
6. 如有需要，再进入策略配置

### 8.2 设计上的依赖关系

设计稿中需要体现以下依赖：

1. 没有 AI 服务提供者时，AI 路由中的目标服务区会缺少可选对象
2. 没有消费者时，AI 路由开启认证后会缺少可选对象
3. 域名是 AI 路由的可选前置资源

### 8.3 页面间入口关系

1. AI 服务提供者管理和 AI 路由管理同属于 AI 服务管理分组
2. 消费者管理作为独立一级入口存在
3. AI 路由列表页需要同时承载“查看调用方式”和“进入策略配置”两个外跳入口
